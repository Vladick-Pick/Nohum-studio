# Obsidian To Repo Doctrine Map

## Intent

Obsidian is the upstream ideation and doctrine-capture layer.

The repository is the executable package layer.

The live Paperclip runtime remains the runtime source of truth after import.

## Mapping Rule

- notes may inspire doctrine, ontology, metrics, or workflow changes
- nothing becomes live policy from Obsidian alone
- a doctrine becomes active only after translation into one or more repo-native artifacts:
  - decision doc
  - operating spec
  - playbook
  - template
  - task
  - role bundle
  - script or deterministic check

## Recommended Flow

1. capture idea or doctrine in Obsidian
2. synthesize it into a concrete repo change proposal
3. translate it into canonical repo artifacts
4. import the updated package into Paperclip runtime
5. treat runtime outputs as the next source of learning

## Rule

Obsidian is for discovery and concept formation.
The repo is for executable organizational design.
