# NoHum Studio Readiness Map

Date: 2026-03-28

This document answers one question only:

What part of the NoHum Studio machine is already real, and what part is still only policy or intent?

## 1. Implemented In Live Runtime

These parts now exist in the live `NOH` company and are actually running:

- one live `Studio company`
- one live `Hypothesis Funnel`
- one live `Studio OS`
- one live venture lane:
  - `Venture - vh-20260328-visual-feedback-agency`
- live agents:
  - `CEO`
  - `Research Lead`
  - `Launch Lead`
- managed instruction files for all three agents:
  - `AGENTS.md`
  - `SOUL.md`
  - `HEARTBEAT.md`
  - `TOOLS.md`
- first research batch completed
- first queue package completed
- reusable Gate B standard published
- reusable launch-readiness checklist published
- first research-machine contract published
- first research storage contract published
- first payment normalization contract published

Verdict:

- the contour is real
- the company is agent-operable
- the first venture lane exists

## 2. Implemented As Durable Operating Artifacts

These pieces are no longer only in conversation. They are saved as canonical docs:

- operating spec
- runtime state
- machine build plan
- research machine v1
- research evidence and storage contract
- first payment signal normalization

Verdict:

- the policy layer is mostly durable
- the artifact layer is now becoming durable

## 3. Partially Implemented

These pieces exist, but are not yet system-enforced:

- `Gate A` logic
- `Gate B` logic
- `venture_id` discipline
- queue-package discipline
- one active + one queued policy
- research quality bar
- payment review state logic

Why only partial:

- the rules exist
- agents can follow them
- but the platform does not yet enforce them automatically

Verdict:

- these parts are operating by instruction and review, not by hard automation

## 4. Not Yet Built As Machine Behavior

These pieces are still missing as real system capability:

- research pod topology in live agent hierarchy
- source registry execution layer
- repeatable traffic / usage validation layer
- repeatable revenue / payment proof validation layer
- `Promote To Venture` routine
- venture document scaffold routine
- `Attach Product Repo` routine
- `Record Payment Signal` routine
- provider-specific payment adapter implementation
- repo scaffold and workspace attach execution path
- repeatable handoff from queue package to venture docs

Verdict:

- this is the real gap between “smart agents with structure” and “machine”

## 5. Current Overall State

Use this language exactly:

- not a prototype fantasy
- not a finished machine
- a live operating contour with a growing durable artifact layer

Practical rating:

- policy completeness: high
- artifact completeness: medium-high
- automation completeness: low
- runtime usability: medium-high
- machine completeness: medium

## 6. What Must Be True Before We Can Say “The Machine Exists”

The machine should only be called real when all of the following are true:

- research runs through a repeatable pod, not one generalist
- source classes and freshness rules are operationalized
- queue packages are always normalized the same way
- venture promotion is repeatable and idempotent
- venture docs are scaffolded without manual invention
- payment events are normalized into a real decision pipeline
- build and launch start from a repeatable venture substrate

Until then, the correct phrase is:

- `NoHum Studio is a live venture OS under construction`
