# Reference: gstack

Date: 2026-03-28

Repository:

- [garrytan/gstack](https://github.com/garrytan/gstack)

## Short Thesis

`gstack` is not primarily an agent company in the NoHum sense.

It is a **workflow operating system for one high-output builder**:

- persistent tools
- strong opinionated skills
- review and QA loops
- design and product challenge skills
- shipping discipline
- local persistence of work artifacts
- session-aware productivity layer

This is why it feels like a “team of specialists” while still being fundamentally a **single-operator control system**.

That distinction matters.

## What gstack Actually Is

At its core, `gstack` combines three things:

### 1. Opinionated workflow skills

The main product is a large skill library:

- `office-hours`
- `plan-ceo-review`
- `plan-eng-review`
- `plan-design-review`
- `review`
- `qa`
- `ship`
- `retro`
- safety skills
- browser skills
- setup skills

This is a role-and-process system more than a multi-agent runtime.

### 2. Persistent tooling substrate

The strongest technical differentiator is not the prompts.

It is the runtime:

- persistent browser daemon
- compiled CLI
- local state files
- cookie/session continuity
- health checks
- version-aware restart
- review and telemetry logging

The repo is serious about tool latency and state continuity.

### 3. Builder ideology encoded into every skill

The same ethos is injected everywhere:

- `Boil the Lake`
- `Search Before Building`
- product-first reframing
- completeness over shortcuts
- extremely opinionated voice and decision style

This gives coherence across skills.

## Structural Model

The most important architectural fact:

- `gstack` is **skill-first**
- `NoHum` is **company-first**

In `gstack`, the virtual “team” mostly appears as:

- specialist skill personas
- chained workflows
- review/readiness dashboards
- user-invoked sequences

In `NoHum`, the system is supposed to appear as:

- real agents
- real reporting lines
- real projects
- real venture stages
- real handoff artifacts
- board approvals

So the similarity is real, but the operating substrate is different.

## The Best Things In gstack

### 1. Extremely strong workflow sequencing

The sequence is explicit:

- think
- plan
- build
- review
- test
- ship
- reflect

This is one of the cleanest parts of the repo.

`gstack` does not leave the user in a blank-prompt state for long.

### 2. Persistent browser as first-class capability

This is a major strength.

The browser is treated as a serious system:

- low latency
- state continuity
- cookie handling
- handoff and resume
- headed mode

This is not toy QA.

### 3. Review and QA are deeply productized

The `review`, `qa`, `ship`, and readiness logging are stronger than what most agent systems have.

Especially useful patterns:

- review-readiness dashboard
- route by phase
- persistent logs under `~/.gstack/projects/`
- explicit cross-skill handoff through saved artifacts

### 4. Shared preamble discipline

Every skill carries:

- session awareness
- telemetry handling
- proactive behavior configuration
- repo-mode handling
- strong AskUserQuestion format

This is very opinionated, but it creates consistency.

### 5. Strong packaging for reuse

The repo is clearly designed to be installed and reused quickly.

That matters because reusable systems win over one-off prompts.

## Weaknesses Relative To NoHum

### 1. Not truly company-native

The system is not built around:

- live company state
- live board approvals
- venture queues
- portfolio logic
- autonomous operating cadence

It is a builder workflow OS, not a venture factory OS.

### 2. “Specialists” are mostly invoked roles, not autonomous employees

This is fine for a solo builder.

But for NoHum, it is not enough.

We need:

- reporting lines
- wake rules
- handoff artifacts
- project containers
- permission boundaries
- queue and gate logic

### 3. Too centered on one human operator

`gstack` assumes a strong central human in the loop.

Even when it talks like a team, it is still optimized around:

- one builder
- one repo
- one workflow stream
- one owner of taste and judgment

NoHum needs a stronger autonomous control plane.

### 4. Not a research machine

It has good planning and product review skills.

It does not give us a venture-factory-grade research operating system:

- no queue-slot economics
- no research pod topology
- no venture lifecycle state model
- no payment-signal contract

## What NoHum Should Borrow

### Borrow hard

- skill discipline as reusable operating modules
- persistent artifact logging
- explicit process stages
- review-readiness style dashboards
- strong browser / QA mentality
- one shared ethos injected consistently
- installability and packaging quality

### Borrow carefully

- “CEO review”, “eng review”, “design review” as concepts

These are useful, but in NoHum they should become:

- real roles
- real gates
- real artifacts

not just prompt modes.

### Do not copy blindly

- one human-centric orchestration model
- giant universal preamble in every role
- everything as slash-skill instead of live org structure
- product-building assumptions where venture-factory assumptions are needed

## What This Means For NoHum

`gstack` validates a few major design decisions we are already making:

### 1. Reusable package matters

Like `gstack`, NoHum should be:

- installable
- portable
- readable
- package-quality

### 2. The machine must encode taste, not only tasks

`gstack` is coherent because its ethos is coherent.

NoHum needs the same thing:

- venture discipline
- one queue slot
- gate discipline
- evidence over narrative
- first payment over feature completeness

### 3. Tooling matters as much as prompts

This is maybe the biggest lesson.

`gstack` feels powerful because the tooling substrate is real.

For NoHum, this means we should continue investing in:

- routines
- venture scaffolds
- payment normalization
- runtime reliability
- artifact persistence

not just in better agent copy.

## Final Judgment

`gstack` is a very strong analogue, but not the same category.

The closest correct description is:

- `gstack` = solo builder workflow OS with specialist skill personas
- `NoHum` = autonomous venture factory OS with real agents, queues, gates, and venture containers

So:

- it is absolutely worth studying
- we should borrow heavily from its workflow rigor and packaging quality
- but we should not let it collapse NoHum into “a better slash-command toolkit”

That would be a downgrade from the company-first model we are building.
