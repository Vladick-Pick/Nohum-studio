# Public Repo Shell Design

## Intent

Reframe `Nohum-studio` from an internal import package README into a public-facing early design project for a minimal-human venture factory.

## Positioning

Primary opening:

> everyone talks about the coming one-person billion-dollar company
> we’re interested in a different question
> what if the real breakthrough is not one person running one giant product, but a minimal-human venture factory systematically operating many small products?

Project framing:

- early open design project
- repo-native operating system design for minimal-human companies
- current implementation target: `Paperclip`
- not the live runtime source of truth
- not a claim of solved autonomy

## Audience

Two entry audiences must both work on the first screen:

- founders and venture-studio operators
- builders, operators, and open-source contributors

## Public Shell

Top-level public shell should contain:

1. `README.md`
   - manifesto-first opening
   - clear explanation of what NoHum is and is not
   - current state and strongest module
   - contribution framing
   - start-here links

2. `MODULES.md`
   - maturity map across major company modules

3. `CONTRIBUTING.md`
   - what kinds of PRs are useful
   - what low-signal contributions to avoid
   - how architecture and governance changes should be proposed

4. `GOVERNANCE.md`
   - repo decision rights
   - boundary between public design work and live runtime decisions
   - review rigor by change type

5. `CODE_OF_CONDUCT.md`
   - concise collaboration rules for a rigorous public project

## Tone

Use a tone that is:

- ambitious
- anti-hype
- systems-oriented
- open to collaboration
- explicit about current incompleteness

## Current-State Message

The repo should present itself as:

- an early open design project
- strongest current module: research
- other modules still draft or in progress
- contributions are welcome on workflows, skills, templates, prompts, evals, governance, and module contracts
